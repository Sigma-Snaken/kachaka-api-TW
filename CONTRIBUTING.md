# kachaka-api 貢獻指南

本指南適用於所有對 kachaka-api 貢獻有興趣的人。我們歡迎各種形式的貢獻，包括錯誤回報、功能建議、程式碼改善、文件修正等。

## 開始使用

對 kachaka-api 的貢獻是讓專案變得更好的寶貴方式。遵循本指南可以提高您的貢獻被順利接受並整合到專案中的可能性。感謝您的合作！

## 貢獻的類型

我們歡迎以下類型的貢獻：

* **錯誤回報**：請回報在 kachaka-api 中發現的錯誤或問題。如果能詳細描述重現步驟和環境資訊將非常有幫助。
* **程式碼改善**：歡迎對現有程式碼的改善、重構、效能提升等相關貢獻。
* **文件修正**：文件的錯字修正、內容改善、補充缺少的資訊等文件相關貢獻也很重要。
* **功能建議**：請提出新功能或改善方案。如果能說明具體的使用案例和預期效果，將更容易進行討論。

## 貢獻的步驟

### 錯誤回報和功能建議

1.  **建立和確認 Issue**：
    * 如需進行錯誤回報或功能建議，請在 GitHub Issues 中建立 Issue。
    * 請確認是否已存在相同的 Issue，如果有類似的 Issue，請加入該討論。
    * 建立 Issue 時，請明確且具體地描述問題或建議的內容。

### 程式碼貢獻

1.  **Fork 儲存庫**：請將 GitHub（或其他平台）上的專案儲存庫 Fork 到您的個人帳號。
1.  **建立分支**：在開始作業前，請建立新的分支。分支名稱如果能反映相關的 Issue 編號或變更內容會更容易理解。
    ```bash
    git checkout -b feature-xyz-issue-123
    ```
1.  **變更程式碼並提交**：變更程式碼並提交。提交訊息應簡潔明瞭地描述變更內容，但由於 kachaka-api 儲存庫採用 Squash Merge，因此提交訊息的詳細程度並不太重要。
1.  **推送**：將本機的變更推送到遠端儲存庫（您的 Fork）。
    ```bash
    git push origin feature-xyz-issue-123
    ```
1.  **建立 Pull Request**：在 GitHub（或其他平台）上，從您 Fork 的儲存庫向原始儲存庫的 `main`（或開發分支）建立 Pull Request。
    * **重要**：在建立 Pull Request 之前，請執行以下命令檢查程式碼的風格和品質。
        ```bash
        ./tools/lint/run_docker.sh
        ```
        此腳本會執行 linter，確認程式碼是否符合專案的規範。加上 `-i` 選項可以自動修正可修正的項目。如果出現錯誤或警告，請修正後再建立 Pull Request。
    * Pull Request 的標題請用英文簡潔描述變更內容。
    * Pull Request 的說明欄請詳細描述變更的目的、背景、相關的 Issue 編號等。此處可以使用日文。
1.  **程式碼審查**：您的 Pull Request 將由專案維護者進行審查。如果有回饋意見，請據此修改程式碼並重新推送。必要時，請再次執行 linter。
1.  **合併**：審查完成並獲得批准後，您的 Pull Request 將以 Squash Merge 方式合併到 `main` 分支。

## 編碼規範

請確認 linter 檢查通過，以及檔案內使用一致的風格。
